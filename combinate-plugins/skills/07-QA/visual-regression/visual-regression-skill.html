Visual Regression Testing

Automated visual comparison between staging environments

Combinate | April 2026

Overview

What It Does

Crawl All Pages

Automatically finds and visits every public page — up to 100 pages per run, at desktop and mobile.

Registration Testing

Automates the full sign-up form, screenshots every step, and validates post-registration behaviour.

Portal Page Testing

Logs in with test accounts and crawls all authenticated pages behind the login gate.

Screenshot Everything

Full-page screenshots at desktop (1440px) and mobile (375px) on both environments for every page.

Pixel-Diff Report

Compares every screenshot pair and produces a branded 2-tab HTML report with diff bars and 404 detection.

Functional QA

Checks broken links, broken images, JS console errors, form validation, and HTTP status on every page.

Use Cases

When to Use It

  • Comparing two staging environments before a client review (e.g. v110 vs v111)
  • Checking staging vs production before a go-live
  • After a large deployment to catch unintended visual changes
  • Any time you need to quickly check what visually changed between two versions

Staging vs Staging

Most common use. Compare two sprint versions side by side before QA sign-off.

Staging vs Production

Final check before go-live. Confirms only intended changes are deployed.

Getting Started

How to Trigger It

Type the slash command in Claude Code with two URLs:

# Basic — previous URL and new URL
/visual-regression [previous-url] [new-url]

# With a custom output folder
/visual-regression [previous-url] [new-url] [output-dir]

/visual-regression https://site-v110.staging.com https://site-v111.staging.com

/visual-regression https://staging.mysite.com https://mysite.com /tmp/regression-prod

Setup

Prerequisites

Run these once per machine. You only need to do this one time.

pip3 install playwright Pillow
python3 -m playwright install chromium

Already set up?

If someone on your machine has already run these commands, you can skip this step and go straight to running the skill.

Test Coverage

What Gets Tested

  • Public pages — Home, About, Features, Products, News, FAQs, Contact
  • Sign-up flow — every step of the registration form, including error states
  • Sign-in flow — valid login, invalid credentials, post-login redirect
  • Portal / dashboard pages — My Details, Order History, Overview, Company
  • System pages — Privacy Policy, Terms, Forgot Password
  • Error & confirmation pages — 404, thank you pages, post-form redirects

Quality Assurance

Functional QA Sweep

Links & Navigation

All internal links, nav menus, and footer links checked for 404s and broken paths.

Broken Images

Every image src is verified. Broken or missing images are flagged in the QA report.

Form Validation

Required fields, invalid inputs, and error messages all tested. Empty submissions are screenshotted.

JS Console Errors

JavaScript errors thrown during page load are captured and listed per page in the QA tab.

HTTP Status Sweep

Every crawled URL is checked for 4xx and 5xx responses on both environments.

Page Performance

Load times are recorded per page. Pages that become 2x slower on the new env are flagged.

Output

What the Report Shows

Tab 1 — Regression Report

One combined table. Rows grouped by section: Public Pages, Sign-up Flow, Sign-in Flow, Portal Pages. Each row shows HTTP status, URLs, diff bars, and screenshot links.

Tab 2 — Functional QA

Structured findings table with Severity, Area, Affects, and Finding columns. Covers broken links, images, JS errors, form issues, and 404s.

Table columns (Regression tab):

  • Previous env — HTTP badge + page URL
  • New env — HTTP badge + page URL
  • Status badge (Changed / Same / 404)
  • Diff bars — desktop % and mobile %
  • Screenshot links — Diff, Old, New (D + M)

All screenshots in one folder. Zip and share — no broken links.

Best Practices

Tips for Better Results

Match seed data first

Confirm both environments are seeded from the same data snapshot before running. Different products or content cause false positives.

Re-run flaky diffs

Dynamic content (carousels, timestamps, live data) causes noise. Re-screenshot any page with an unexpectedly high diff before flagging it as changed.

Note credentials in Teamwork

If the project requires admin activation of test accounts, save the test emails in a Teamwork task comment so the team can activate them in the admin panel.

Review highest diffs first

Sort the report by diff percentage. Pages sharing the same template (e.g. product filter variants) will all appear — focus on the highest % first to cut through the noise.
